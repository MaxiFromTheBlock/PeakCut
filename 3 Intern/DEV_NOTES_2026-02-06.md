## Probleme die wir gefunden haben

### 1. Video-Offset wurde berechnet aber nie angewendet
- `session.video_offsets` wurde via Cross-Correlation berechnet
- `session._offset_lookup_ms` existierte
- Aber beim Seek/Play wurde der Offset NIE auf die Video-Position angewendet
- Fix: Conversion-Methoden `_mix_to_video_ms()` und `_video_to_mix_ms()` in video_preview_peak.py

### 2. Screenshot + Analyse Konflikt
- MoviePy nutzt intern `multiprocessing`
- Screenshot nutzt ffmpeg subprocess + PIL + LUT in QThread
- Beides zusammen → Crashes (Exit code 139, Segfault)
- Erste Vermutung: Zwei QMediaPlayers
- Tatsächliches Problem: Ressourcen-Konflikte zwischen Prozessen

### 3. LUTProcessor nicht Thread-safe
- Gleiche LUTProcessor-Instanz wurde zwischen Main Thread und LUTWorker geteilt
- Race Condition beim Zugriff auf numpy Arrays
- Fix: LUTWorker bekommt eigenen LUTProcessor

### 4. QVideoFrame Lifetime Problem
- `QVideoFrame` aus `videoFrameChanged` Signal ist nur während Callback gültig
- Wir haben `self._last_frame = frame` gespeichert
- Später Zugriff → Crash (Speicher bereits freigegeben)
- Fix: `self._last_frame_image = frame.toImage().copy()`

### 5. VideoFileClip Memory Leak
- `clip = VideoFileClip(video_path)` wurde nie geschlossen
- File Handles und ffmpeg-Prozesse blieben offen
- Fix: `try/finally` mit `clip.close()`

### 6. Cross-Correlation Memory Explosion
- `correlate(target, reference, mode='full')` für lange Audio-Dateien
- 1 Stunde @ 48kHz = 346 Millionen Samples × 8 Bytes = 2.7 GB RAM
- Fix: Downsampling vor Correlation (Faktor 10)

### 7. Subprocess Deadlock
- AnalysisWorker las nur stderr
- stdout füllte sich → Buffer voll → Subprocess blockiert
- Fix: Separater Thread für stdout-Reading

### 8. MoviePy stdout Pollution
- MoviePy schrieb Progress nach stdout statt stderr
- Vermischte sich mit JSON-Output
- JSON Parser failed: "Expecting value: line 1 column 1"
- Fix: `logger=None` bei `write_audiofile()`

### 9. numpy int64 nicht JSON-serialisierbar
- Peak-Positionen waren numpy.int64
- JSON serialization failed
- Fix: `int(t)` bei der Konvertierung

### 10. Audio wurde nie geladen (nach Subprocess-Umbau)
- Analyse lief in separatem Prozess
- Nur JSON-Results kamen zurück (peaks, offsets)
- AudioSegments wurden nicht geladen
- Fix: `session.load_audio_lazy()` vor Export und Playback

### 11. Kein Audio bei Video-Preview
- Video war muted (by design)
- Aber auch kein Mix-Audio wurde abgespielt
- `session.play_current()` wurde nur ohne Videos aufgerufen
- Fix: Immer `session.play_current()` aufrufen

### 12. Audio nicht synchron mit Video
- Video springt zu offset-korrigierter Position
- Audio spielt von Anfang des Clips
- Keine echte Synchronisation implementiert

## Vermutungen / Theorien

### Warum so viele Crashes?
Die App wurde inkrementell erweitert ohne die Architektur anzupassen:
- Video-Preview kam dazu (QMediaPlayer + QVideoSink)
- LUT-Processing kam dazu (numpy in Thread)
- Screenshots kamen dazu (ffmpeg + PIL + LUT)
- Clip-Editing kam dazu (mehr State, mehr UI)
- Analyse blieb im Main Process (MoviePy mit multiprocessing)

Alles zusammen in einem Prozess mit geteiltem State → Race Conditions, Memory Issues, Deadlocks.

### Was wäre die saubere Architektur?
1. **Analyse** - Komplett separater Prozess (haben wir jetzt)
2. **Video-Preview** - Nur Video anzeigen, kein Audio
3. **Audio-Playback** - Separates System (simpleaudio ist ok)
4. **Screenshots** - Komplett in ffmpeg (haben wir jetzt)
5. **State** - Klar definiert, nicht zwischen Threads geteilt

### Was fehlt für echten Audio-Video Sync?
- Video-Position tracken (QMediaPlayer.positionChanged)
- Audio-Segment zur gleichen Zeit starten
- Oder: Audio komplett über QMediaPlayer (aber dann zwei Player-Problem)
- Oder: Akzeptieren dass es nicht perfekt synchron ist
